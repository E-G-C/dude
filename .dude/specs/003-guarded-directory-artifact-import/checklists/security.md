# Security Checklist: Guarded Directory Artifact Import

This checklist is advisory acceptance evidence, not a second execution board.

- [ ] Local traversal inventories every directory, regular file, link, and non-regular entry without following links; rejects escapes and normalized/case collisions; and detects observed root, ancestor, type, size, identity, and content drift.
- [ ] The fixed bounds are enforced before planning and while reading: depth 12, 256 entries, 128 regular files, 1 MiB per file, 4 MiB total, bounded metadata/raw responses, and 30 seconds per request.
- [ ] GitHub accepts only canonical public tree URLs, resolves the requested ref to an immutable commit through bounded API metadata, constructs commit-pinned `raw.githubusercontent.com` URLs, refuses redirects and response-provided URLs, and verifies Git blob SHA plus SHA-256.
- [ ] Every bounded entry appears in stable order; one root owns all descendants, multiple roots use unique nearest-root ownership, and only selected-root license/notice files may be shared.
- [ ] Every entrypoint has strict present frontmatter with scalar `name` and `description`; skills also pass the current `normalizeSkillDir`, while `model`, `compatibility`, `tools`, `license`, and any other focused-import adaptation metadata Block with file-specific focused-import or clean-source guidance.
- [ ] Every agent entrypoint has exactly one unfenced, unprefixed, standalone physical body line equal to the complete current team-expansion `**Coordinator-only artifacts:**` paragraph. The deterministic line check distinguishes missing, malformed, duplicated, and noncanonical blocks and emits the corresponding file-specific blocking diagnostic with focused single-file import/adaptation or clean-source guidance.
- [ ] Every support, example, script, license, notice, and agent-entrypoint byte is unchanged; only a skill entrypoint's parsed `name` scalar changes to the final `dude-local-*` directory, records sole transform `rewrite-skill-name`, and preserves all other bytes and line endings. Every other output has empty `transform_ids`.
- [ ] Exact LF and CRLF agent/skill frontmatter outputs pass current Dude lint without normalization; end-to-end regressions cover one accepted unchanged canonical agent plus blocked missing/malformed boundaries, while focused lint regressions include namespaced skill-name equality after `rewrite-skill-name`, retain mismatch refusal, and prove that the shared narrow reader still rejects bare CR and malformed delimiter shapes without becoming a general YAML parser.
- [ ] Fixed mapping detects literal relative links or references that would break and Blocks with exact evidence; no ownership is inferred from references, no content is rewritten, and invalid placement Blocks instead of triggering automatic repair.
- [ ] Every ambiguous/unowned ownership, malformed/adaptation-sensitive entrypoint, missing/malformed/duplicated/noncanonical agent boundary, broken reference, unsafe output/ancestor, destination collision, source/output overlap, path alias, and source/destination file-identity failure appears in required `blocking_diagnostics` with a stable code, normalized principal path or null, sorted unique related paths, concise message, and actionable guidance when possible; identities and order are deterministic, invalid records reject, and no file is silently excluded.
- [ ] The one published static rule table scans every regular file as bytes or strict text; only tightly bound high-confidence constructs with dangerous operands joined in one expression or command Block, mere same-file coincidence cannot Block, broad indicators Warn, and every finding retains path, rule, evidence, location, category, severity, and explanation.
- [ ] Static scanning performs no Markdown, JavaScript, shell, or other context parsing and no context-based severity reduction; broad indicators in prose, examples, comments, or executable content remain Warned.
- [ ] Opaque files are scanned for byte signatures and printable evidence and cannot produce Clean unless v1 later defines an explicit review path; no binary content is declared safe by type alone.
- [ ] Analysis retains one canonical entries list, minimal groups, exact outputs, static findings/diagnostics, and batches of at most 16 complete text files and 256 KiB; removed counts, ownership roles, source hashes, byte counts, and coverage paths reconstruct exactly from retained data and fixed rules.
- [ ] Optional review input contains only schema identity, `analysis_sha256`, sorted unique `reviewed_batch_ids`, and advisory findings naming one reviewed batch and valid path; malformed, stale, unknown, duplicate, or inconsistent IDs/findings reject, while omitted IDs, over-budget text, and opaque files remain derivably uncovered and force Warned.
- [ ] Language-model review sees exact complete batches, adds only `info` or `warn` purpose/intent/cross-file/prompt-manipulation findings, and cannot Block, authorize, normalize, erase, or downgrade deterministic evidence. Analysis and batch content are treated as potentially sensitive and carry no safety-proof claim.
- [ ] `static_decision` reconstructs from static findings and `blocking_diagnostics`; every diagnostic forces Blocked and appears in the complete preview. `plan-directory` requests no confirmation. Clean apply accepts only `confirm-import`; Warned apply accepts only `confirm-warned-import:<plan_sha256>`; Blocked has no plan, force, override, or accepted literal.
- [ ] `replace_paths` equals every destination-sorted output with an existing regular-file state, and the applicable literal authorizes that complete set with no selective replacement, merge, auto-rename, or implicit overwrite.
- [ ] Planning recomputes source, ownership, static evidence, blocking diagnostics, exact outputs, simple destination states, replacements, and source/output overlap without accepting confirmation, creating transaction material, or mutating a destination.
- [ ] `plan_sha256` omits only its own field; the warned literal is rendered after hashing and is not embedded in the payload, so apply can validate plan-bound confirmation non-circularly.
- [ ] Planning and apply reject lexical, canonical, case, ancestor/descendant, symbolic-link, mount, pairwise-output, alias, and source/destination file-identity overlap before mutation.
- [ ] Apply validates the plan and exact derived literal once, repeats all deterministic source/output/destination checks, derives a safe disjoint same-filesystem transaction parent where feasible, and rechecks transaction overlap in a separately tested nonmutating preflight.
- [ ] Staging and backups remain outside the source and on the destination filesystem when feasible, are verified before writes, and refuse safely when no suitable transaction location exists.
- [ ] Failure injection at staging, backup, final preflight, every write, verification, rollback, and cleanup proves exact installation, verified restoration, or `recovery-failed` with every uncertain path and retained recovery material.
- [ ] Analysis, planning, and apply execute no imported file, hook, interpreter, package manager, dependency installer, transitive fetch, or automatic activation.
- [ ] Existing single-file import sources, confirmation, naming, safety checks, destination binding, and honest partial-write reporting remain covered without a false transactionality claim.
- [ ] Source and generated guidance describe exactly three commands, two persisted artifacts, fixed limits, ownership and byte rules, clean-entrypoint and broken-reference blocks, scanner boundary without context suppression, review coverage truth, plan-bound literals, complete replacements, overlap refusal, rollback limits, and the absence of malware-proof safety claims.
