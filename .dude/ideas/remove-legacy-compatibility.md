# Idea: Remove Legacy Compatibility From The Bundle

## Idea

The bundle has grown bigger and less focused, and I want the next optimization effort to start by removing its legacy baggage. Backward-compatibility behavior and machinery for migrating old bundle or workspace layouts add active prompt and token cost, complexity, and maintenance work that I no longer need.

I am effectively the only consumer, and I will use the current canonical bundle format from now on. Older installations do not need an in-bundle migration path; anything still worth migrating can be handled manually or separately outside the active bundle. This is the first bounded cleanup slice, not the entire optimization roadmap: remove what is shipped solely for old-layout migration or backward compatibility so the bundle becomes smaller, simpler, and cheaper to carry.

After that legacy cleanup, the second slice should optimize wording throughout the bundle so each instruction says what it needs to with the fewest practical words. The goal is lower active token and context cost, not merely fewer repository bytes. This should preserve focus, goals, behavioral precision, safety constraints, routing reliability, and readability; shortening that drops essential meaning is not an optimization.

As another phase-two optimization idea, I want to inventory procedures currently owned by prompts and find the deterministic mechanics that can follow the same script-and-package pattern as the current implementation. Stable parsing, inventory, validation, calculation, rendering, and guarded mutations can move into dependency-light, tested scripts beside the owning core skill or optional pack, while agents keep semantic interpretation, routing, prioritization, approvals, and quality judgment. Read and preview paths should stay non-mutating and machine-readable by default, and writes should use the existing guarded, repeatable safety patterns.

I want Dude to contribute concrete optimization ideas for that phase. Broader optimization should continue only after measuring what the first two slices actually change.

## Normalized Intent

1. Remove legacy-only migration and backward-compatibility machinery while preserving current-format maintenance.
2. Minimize wording and active context throughout the bundle while preserving behavior, focus, safety, routing reliability, and readability; as a phase-two strategy, inventory prompt-owned procedures and extract only deterministic mechanics into canonical scripts or packages.
3. Continue broader optimization only after measuring the effects of the first two slices.

## Constraints

- Keep legacy removal as phase one and wording/context optimization as phase two; defer broader optimization until their effects are measured.
- Meaning, focus, goals, behavioral precision, safety, routing reliability, and readability take priority over raw token or byte reduction; do not blindly minify.
- Preserve all current canonical project data and active work.
- Preserve maintenance capabilities for current-format bundle upgrades and current-format pack composition unless the user later changes the scope.
- Script extraction must reduce active prompt/context and duplication rather than merely relocate complexity; give each mechanic one canonical owner and extend existing mechanics instead of creating a second implementation.

## Assumptions

- Unless overridden, the current canonical format (`.dude/ideas/`, `.dude/specs/`, and the current metadata/profile shape) will be the only supported format after this cleanup.
- Unless overridden, remove the deprecated `@dude draft` compatibility alias; `@dude brainstorm` remains the supported intake action.
- Unless overridden, remove in-bundle workspace, layout, and profile migration commands, skills, engine paths, diagnostics, compatibility parsing, tests, and documentation that exist only for old structures.
- Unless overridden, preserve normal upgrades between current-format bundle versions and current-format pack composition because they are maintenance capabilities, not old-layout migration.
- Unless overridden, preserve all current canonical project data and active work.
- Breaking old layouts is accepted; no automatic fallback, in-bundle migration path, or compatibility guarantee is required.

Dude-proposed defaults for phase two (the user may override them):

- Measure baseline and post-change token and character footprints for the always-loaded coordinator and instructions plus common workflows; optimize active context before total repository bytes.
- Deduplicate repeated rules across the coordinator, shared instructions, skills, agents, and docs; keep one canonical owner and use short references elsewhere where runtime discovery permits.
- Move rare or specialized procedures out of always-loaded coordinator and instruction surfaces into on-demand skills or optional packs.
- Replace repeated explanatory prose and near-identical examples with concise imperative rules and one representative example.
- Consolidate overlapping rules and sections, and remove contradictory or obsolete guidance.
- Shorten skill descriptions and frontmatter trigger text while preserving routing accuracy.
- Preserve critical safety gates, ownership boundaries, confirmation tokens, and executable behavioral contracts.
- Use focused regression tests and source/generated parity checks to prove wording reductions did not change behavior.
- Measure before choosing a numeric token-reduction target; do not invent a percentage yet.
- During `define`, consider separate implementation packages for phase one and phase two while keeping them in this single optimization roadmap idea.

### Dude-proposed deterministic automation candidates (the user may override them):

Existing pattern: `board.mjs`, `lint.mjs`, `compose.mjs`, `upgrade.mjs`, `memory.mjs`, scaffolding, Beads, and build/release tooling already separate deterministic mechanics from agent judgment. Follow that contract: colocate each dependency-light script and its `node:test` coverage with one owning core skill or optional pack; default inspect/plan to non-mutating JSON with explicit failure states; fail closed; use expected hashes or digest-bound plan/apply plus atomic, idempotent writes for mutations; keep prompts focused on invocation, authority, and decisions.

Prioritized candidates:

1. **Core canonical companion/feature resolver** - `feature.mjs resolve/list-defined` for repeated exact `spec_path` ownership logic; this is the smallest recommended first extraction after phase-one deletion.
2. **Core read-only workflow collector** - `workflow.mjs inspect/diff/self-check` for ownership, task counts and readiness, fence/breadcrumb validity, board staleness, snapshot drift, and log tails; the coordinator still chooses lanes and recommendations.
3. **Core idea-ledger editor/log appender** - `idea.mjs inspect/plan-update/apply/append-log`, preserving user sections byte-for-byte with hashes and event IDs.
4. **Core task reconciliation extension** - extend `tasks.mjs` or `board.mjs` to compute exact durable-key reconciliation; semantic mappings and destructive history decisions still require agent or user input.
5. **Core definition allocation/owner-binding transaction** - `definition.mjs plan/apply` for monotonic package numbering, collision checks, staged artifacts, exact owner binding, rollback, and no-op replay; Spec Lead still authors semantic content.
6. **Optional Beads-pack mirror** - extend `beads.mjs` with status/sync plan/apply bound to complete Beads and task-file inventories.
7. **Optional authoring-pack prompt audit** - read-only, tokenizer-pinned `prompt-audit.mjs` for active-context/token footprint and duplicate-rule auditing, with no automatic semantic deduplication or deletion.
8. **Partial core roster advisory** - `roster.mjs inventory/rank` for lexical ranking only; final specialist selection, decomposition, and authority stay with Dude.

Judgment exclusions:

- Do not script guardrail inference or independent review/approval; only their structural checks or accepted-text persistence are deterministic.
- Do not automate or package legacy migration/profile reconciliation, deprecated `draft`, legacy diagnostics, legacy identity kinds or paths, or obsolete cleanup. Phase one deletes them; reuse only their general safety patterns.
