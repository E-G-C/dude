# Idea: Guarded Directory Artifact Import

## Idea

Allow the user to ask Dude to import skills, agents, and related artifact components by pointing to a root local directory or a GitHub (or equivalent ) directory/root URL rather than listing individual files. 

Example input: `https://github.com/addyosmani/agent-skills/tree/main/skills/idea-refine`.

- Discover and import subsequent relevant artifacts under that root, including skills, agents, examples, framework or support files, and scripts that belong to the imported artifact.
- When imported skills or agents invoke or include scripts or other executable or supporting content, inspect the discovered contents for security threats and suspicious intent before installation or later invocation.
- Destructive actions, suspicious prompt instructions, prompt-injection behavior, unsafe command execution, credential or data access or exfiltration, network activity, privilege escalation, persistence, obfuscation, or other concerning behavior should trigger a clear warning to the user before import or installation.
- The warning should identify which artifact or file is suspicious and what behavior is concerning so the user can make an informed decision.
- Security review is risk detection and warning, not a claim that semantic scanning can prove safety.
- Existing single-file import safety, including preview and confirmation, no silent writes, namespacing, and path, size, and transport constraints, should not be weakened.
- Be pragmatic: implement an appropriate, useful security and directory-import boundary without over-engineering, excessive infrastructure, or pretending to solve malware detection completely. Balance implementation effort with meaningful protection.
- Always identify deterministic process during this implementation and implement those as scripts rather than delegating everything to the LM.
- Treat this as one cohesive guarded directory-import outcome, not a roadmap split.

## Open Questions

1. Should directory import default to importing the entire bounded subtree, or present an inventory and let the user select artifacts or files before confirmation?
   Answer: Yes, it should default to importing the entire subtree.
2. For suspicious executable or prompt content, should the first version hard-block only clear high-risk patterns and otherwise warn and require explicit confirmation, or warn only for all findings?
   Answer: Yes, block and warn accordingly.
3. Should v1 support canonical GitHub `tree/<ref>/<path>` URLs plus local directories only, deferring arbitrary Git hosts and repositories?
   Answer: I would say yes local and github for now depending. It could be implemented in a way that is extensible for other git hosts in the future. So any code or script should be extensible to other providers. For now let's go with these two.

## Normalized Intent

- Accept one bounded local directory or canonical public GitHub `tree/<ref>/<path>` URL through a fixed dispatcher and treat the complete selected subtree as one cohesive import.
- Let one valid agent or skill root own all descendants; with multiple roots use unique nearest-root ownership, share only selected-root license/notice files, and Block ambiguous or unowned regular files with narrower-root guidance.
- Preserve every output byte exactly except one strict-parser replacement of a skill entrypoint's `name` scalar with its final `dude-local-*` directory; Block entrypoints needing focused-import adaptation and literal relative references broken by fixed mapping.
- Scan every bounded regular file with one small published byte/text indicator table: only tightly bound high-confidence dangerous constructs Block, every broader indicator Warns, and no context parser reduces severity.
- Allow optional language-model review only over exact complete-file batches of at most 16 files and 256 KiB, represented by reviewed batch IDs plus advisory findings; uncovered, over-budget, or opaque content forces Warned.
- Expose exactly `analyze-directory`, read-only confirmation-free `plan-directory`, and `apply-directory`, producing one analysis and one final reviewed plan; apply requires `confirm-import` for Clean, `confirm-warned-import:<plan_sha256>` for Warned, and has no path for Blocked.
- Keep deterministic discovery, normalization, hashing, ownership, scanning, plan/preflight validation, and rollback-backed apply mechanics in tested scripts.

## Constraints

- Fix v1 at depth 12, 256 entries, 128 regular files, 1 MiB per file, 4 MiB total, bounded metadata/raw responses, and a 30-second request timeout.
- Use GitHub API only for immutable commit and bounded tree metadata; construct commit-pinned raw URLs, refuse redirects and response-provided URLs, and verify Git blob SHA plus SHA-256.
- Reject every local source/output overlap or alias and every source/destination regular-file identity overlap; derive and recheck a safe transaction parent during apply, with staging outside source and on the destination filesystem when feasible.
- Preview every bounded entry and file-specific finding; do not silently exclude files, infer ownership from references, rewrite references or code, or automatically repair content.
- Bind planning and apply to exact source bytes, ownership, findings, outputs, simple missing/regular-file destination states, and the complete replacement set; bind warned confirmation non-circularly to the final plan hash and recompute deterministic facts before planning and apply.
- Preserve existing single-file import behavior and protections unchanged, including source forms, confirmation, naming, destination binding, and honest result semantics.
- Perform no imported execution, package or dependency installation, transitive fetching, arbitrary-host access, authentication/private-repository flow, sandbox claim, plugin loading, or malware-proof safety claim.
